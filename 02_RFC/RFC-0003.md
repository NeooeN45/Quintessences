## Objet

Proposer et documenter les principes fondamentaux de l'architecture
distribuée de GSIE — baptisée **GSIE-Net** — pour conservation et
traçabilité en vue de la Phase 2 (Architecture).

Ce RFC ne produit pas de code. Il capture la vision architecturale du
Fondateur pour qu'elle ne soit pas perdue et qu'elle oriente les
travaux d'architecture future.

---

## Motivation

La vision de GSIE dépasse celle d'une application mobile ou d'un
logiciel SIG classique. GSIE est un **écosystème distribué** :
un réseau de nœuds intelligents capables de fonctionner de façon
autonome, de se synchroniser opportunément et de collaborer autour
d'une base de connaissances commune.

Les principes suivants, formulés par le Fondateur, constituent la
fondation architecturale à documenter et à préserver.

---

## Principes fondateurs de GSIE-Net

### 1. Offline-First par nature

Le fonctionnement hors-ligne n'est pas une option dégradée — c'est
l'état normal d'un technicien de terrain. GSIE doit fonctionner
identiquement avec ou sans réseau.

> « Le technicien ne doit pas se rendre compte qu'il n'a plus Internet. »

Chaque nœud terminal (téléphone) embarque localement :
- les cartographies nécessaires à la mission ;
- les données de la parcelle (historique, inventaires, plan de gestion) ;
- les bases de connaissances scientifiques (essences, protocoles) ;
- les modèles d'IA pour les tâches courantes ;
- les guides et documents de référence (ONF, protocoles, clés de détermination).

La synchronisation se fait lors d'une fenêtre de connectivité, sans
action de l'utilisateur.

---

### 2. Architecture en couches séparées (GSIE-Net Stack)

L'architecture réseau de GSIE est organisée en couches indépendantes,
à l'image du modèle OSI :

```
┌──────────────────────────────────────┐
│  Couche métier GSIE                  │  Inventaire, martelage, diagnostic,
│  (moteurs, applications, IA)         │  cartographie, protocoles
├──────────────────────────────────────┤
│  Couche GSIE-Net                     │  Routage, synchronisation, découverte
│  (logique de réseau GSIE)            │  des nœuds, sécurité, priorités
├──────────────────────────────────────┤
│  Couche transport                    │  Meshtastic (Phase 2), LoRa, Wi-Fi,
│  (protocoles de transmission)        │  Bluetooth, Ethernet, 4G/5G, satellite
├──────────────────────────────────────┤
│  Couche physique                     │  Modules radio, antennes, capteurs,
│  (matériel)                          │  relais, smartphones, serveurs
└──────────────────────────────────────┘
```

**Conséquence clé :** si une technologie radio est remplacée dans 10 ans,
seule la couche transport change. La logique métier et les données
restent intactes.

---

### 3. Topologie de réseau — trois niveaux

**Niveau 1 — Terrain (nœuds terminaux)**

Chaque technicien dispose d'un smartphone avec l'application GSIE et,
optionnellement, d'un module GSIE Radio (XIAO + SX1262) connecté en
Bluetooth. Le téléphone est le nœud de collecte. Le module radio est
un modem sans intelligence propre.

**Niveau 2 — Mesh LoRa (réseau maillé)**

Les nœuds se connectent entre eux sans topologie fixe. Le réseau
trouve automatiquement le meilleur chemin :

```
Technicien A → Relais solaire → Relais ONF → Technicien B
Technicien A → Technicien B → Technicien C → Serveur
```

Aucune connexion directe n'est requise. La résilience est inhérente
à la topologie maillée.

**Niveau 3 — Edge Nodes GSIE (relais intelligents)**

Les relais ne sont pas de simples répéteurs. Ce sont des nœuds
intermédiaires capables de :
- stocker temporairement les données en transit ;
- compresser les données avant retransmission ;
- gérer les priorités (données critiques avant données de masse) ;
- éviter les doublons (déduplication) ;
- synchroniser avec le serveur central si une connexion Internet est
  disponible.

---

### 4. Synchronisation orientée données (modèle Git)

GSIE synchronise des **données**, pas des messages. Chaque modification
terrain est un **commit** sur un objet identifié.

Exemple de cycle de vie d'un arbre :

```
Arbre 154 — DBH modifié     → commit local
           — Photo ajoutée  → commit local
           — Martelage       → commit local
           — Synchronisation → merge serveur
```

**Propriétés attendues :**
- Deux techniciens peuvent modifier la même parcelle hors-ligne
  simultanément ; la synchronisation résout les conflits.
- Chaque objet a un identifiant stable permettant la déduplication.
- Le serveur peut demander un objet précis : « envoie-moi l'arbre 154,
  il me manque sa hauteur ».
- Le transfert est différentiel : seul ce qui change est transmis.

---

### 5. Réseau orienté données (pas orienté messages)

La plupart des réseaux radio transmettent des messages : « envoie
ce texte à Paul ». GSIE-Net est orienté **données** : chaque nœud
connaît son catalogue de données et peut répondre à une requête
sémantique.

Exemple de requête :
> « Qui possède les données de la parcelle 27, arbres martelés
> entre 10h15 et 11h00 ? »

Le serveur identifie le nœud qui possède ces données, les demande
directement, les intègre. Aucune intervention humaine.

---

### 6. Intelligence distribuée à plusieurs niveaux

L'IA n'est pas monolithique dans GSIE. Elle est distribuée selon les
capacités du nœud :

| Nœud | Capacités IA |
|---|---|
| Téléphone | Modèle embarqué léger — reconnaissance, aide immédiate |
| Edge Node | Cache de contexte, pré-filtrage |
| Serveur GSIE | Modèles lourds, calculs complexes, apprentissage |

Comportement attendu :
- question simple → réponse locale (téléphone) ;
- question complexe → synchronisation → calcul serveur → retour ;
- le technicien ne voit aucune différence.

Les 14 moteurs GSIE s'articulent sur cette infrastructure :

```
GSIE Moteurs
├── Evidence Engine       — validation des données terrain
├── Knowledge Engine      — connaissances scientifiques locales
├── Reasoning Engine      — raisonnement (local/distant selon complexité)
├── Diagnostic Engine     — diagnostic assisté
├── Recommendation Engine — propositions sylvicoles
├── Validation Engine     — contrôle qualité
├── GIS Engine            — cartographie et localisation
├── Climate Engine        — données climatiques (cache + distant)
├── Pedology Engine       — données pédologiques
├── Botanical Engine      — identification des essences (IA locale)
├── Forest Dynamics Engine— dynamique et croissance
├── Correlation Engine    — croisements de données
├── Learning Engine       — amélioration continue (serveur)
└── Simulation Engine     — projections (serveur)
```

---

### 7. Mission de terrain — préparation automatique

Avant toute mission, le serveur GSIE prépare un **bundle de mission** :
un ensemble cohérent de données téléchargé sur le téléphone avant
le départ sur le terrain. Le technicien accède à tout hors-ligne
sans configuration manuelle.

Contenu typique d'un bundle de mission :
- carte IGN + orthophotos + MNT ;
- limites parcellaires cadastrales ;
- données propriétaire et plan de gestion ;
- inventaires précédents et historique ;
- données LiDAR disponibles ;
- historique climatique local ;
- modèle de croissance des essences présentes ;
- guides ONF, clés de détermination, protocoles ;
- modèle d'IA de reconnaissance d'essences.

---

### 8. Serveur GSIE — chef d'orchestre actif

Le serveur GSIE ne reçoit pas passivement des données. Il pilote
activement la qualité des données collectées :

- il détecte les données manquantes ou incohérentes ;
- il formule des demandes ciblées aux nœuds qui possèdent les données ;
- il arbitre les conflits de synchronisation ;
- il déclenche la mise à jour des bundles de mission.

Exemples de comportements actifs :
> « Technicien 4, il me manque les arbres 152 à 170 de la parcelle 18.
> Renvoie uniquement ces données. »

> « GPS de l'arbre 23 trop imprécis (HDOP > 5). Attends 20 secondes
> et renvoie une nouvelle position. »

> « Photo de l'arbre 87 floue. Une nouvelle photo est nécessaire. »

## Options envisagées

| Option | Description | Avantages | Risques |
|---|---|---|---|
| **A** | Meshtastic comme transport GSIE-Net | Mature, open-source, communauté active | Dépendance externe, protocole non adapté aux besoins GSIE |
| **B** | Protocole GSIE-Net propriétaire sur LoRa | Contrôle total, optimisé pour GSIE | Complexité, temps de développement |
| **C** | Meshtastic Phase 2, protocole GSIE Phase 3+ | Pragmatisme : démarrer sur l'existant | Coût de migration futur |

La recommandation (à débattre en Phase 2) est l'**Option C** : utiliser
Meshtastic comme transport initial, tout en encapsulant la logique
GSIE-Net dans une couche d'abstraction qui permettra une migration
propre vers un protocole dédié.

## Impact et dépendances

- **Phase 2 — Architecture** : ce RFC est le document de référence pour
  l'architecture réseau et distribuée.
- **Article T-8** (Constitution Technique) : ce RFC étend et précise T-8.
- **GSIE/ENGINES/** : les 14 moteurs devront déclarer leur comportement
  hors-ligne et leur niveau de distribution dans leurs fiches Phase 2.
- **Article T-2** (couplage faible) : GSIE-Net est lui-même un contrat
  d'interface entre la couche transport et la couche métier.

---

## Ce que ce RFC ne fait PAS

- Il n'implémente aucun code.
- Il ne définit pas les protocoles en détail (Phase 2).
- Il ne choisit pas de technologie radio définitive.
- Il ne contredit aucun article constitutionnel.
