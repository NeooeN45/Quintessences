# DEC-000030 — Adoption de RFC-0018 : identification botanique assistée (Pl@ntNet)

| Champ | Valeur |
|---|---|
| **ID** | DEC-000030 |
| **Statut** | Validated |
| **Date** | 2026-07-20 |
| **Décideur** | Camille Perraudeau (Fondateur) |
| **RFC liée** | RFC-0018 (Identification botanique assistée Pl@ntNet et extension du Botanical Engine) |
| **RFC complémentaires** | RFC-0016 (schéma forestier — `AutecologyProfile`), RFC-0015 (registre de modèles/licences), RFC-0017 (veille d'origine) |
| **Spécification liée** | `05_SPECIFICATIONS/GEOSYLVA/GEO_004_IDENTIFICATION_BOTANIQUE_PLANTNET.md` |

## Décision

Valider la RFC-0018 et autoriser le développement du **volet en ligne
uniquement** (§5 de la RFC) :

1. Le modèle de données de traçabilité (§5.1) :
   `BotanicalIdentificationRequest`, `BotanicalIdentificationResult`,
   `BotanicalIdentificationDecision`, avec le statut à trois valeurs
   `SUGGESTION_IA` / `VALIDEE_UTILISATEUR` / `REJETEE` — jamais
   fusionné avec une observation directe.
2. Le connecteur de validation vers `AutecologyProfile` (RFC-0016),
   sur le modèle de `engines/botanical/extraction_bridge.py` —
   écriture uniquement sur décision `VALIDEE_UTILISATEUR`.
3. La chaîne applicative GeoSylva → serveur GSIE → Pl@ntNet →
   normalisation → validation humaine (§5.2), clé API exclusivement
   côté serveur, retrait des métadonnées GPS avant envoi.
4. Une implémentation par tranches verticales, sur le modèle déjà
   éprouvé de RFC-0016 : Tranche 1 = schéma de données seul (aucun
   appel réseau Pl@ntNet), tranches suivantes = client Pl@ntNet côté
   serveur, puis intégration app mobile GeoSylva.

Le **volet modèle embarqué offline** (§6 de la RFC) reste explicitement
**hors périmètre d'implémentation** de cette décision — à l'étude
seulement, aucun jalon de réalisation ouvert.

## Contexte

RFC-0018 a été rédigée le 2026-07-20 (issue de DEC-000029, scission de
RFC-0017) puis complétée d'une spécification fonctionnelle (GEO-004)
le même jour. Le fondateur a priorisé ce RFC comme premier chantier
d'exécution de la veille Pl@ntNet/NVIDIA, avant RFC-0019.

## Impact

- **Code** : nouveau module `gsie_api.infrastructure.models.identification`
  (Tranche 1), extension de `resources/validators.py`,
  `infrastructure/models/enums.py`.
- **Architecture** : 3 nouvelles entités satellite du métamodèle,
  aucune modification du schéma forestier RFC-0016 (extension par
  connecteur, pas par duplication).
- **Mémoire** : mise à jour `PROJECT_MEMORY.md`, `CHANGELOG.md`.
- **Hors périmètre de cette décision** : client réseau Pl@ntNet,
  routes serveur, UI GeoSylva, retrait EXIF GPS effectif — tranches
  ultérieures, chacune pouvant nécessiter une confirmation
  intermédiaire du fondateur avant d'engager un appel réseau réel vers
  un service tiers.

## Alternatives rejetées

1. **Développer le client Pl@ntNet et l'UI GeoSylva en un seul bloc** —
   rejeté : risque de découvrir un problème de licence/contrat en
   cours de route après avoir déjà écrit du code d'intégration réseau ;
   le découpage en tranches permet de vérifier la confirmation écrite
   Pl@ntNet (préalable §8 de la RFC) avant d'aller plus loin que le
   schéma de données.
2. **Fusionner `BotanicalIdentificationResult` dans `AutecologyProfile`
   directement** — rejeté : `AutecologyProfile` porte une observation
   autécologique validée (RFC-0016), pas une suggestion en attente ;
   les fusionner romprait la distinction `SUGGESTION_IA` vs `observé`.

## Risques

- Aller trop loin dans l'implémentation avant la confirmation écrite
  de Pl@ntNet sur les conditions commerciales — mitigé en limitant
  cette décision au schéma de données (Tranche 1), sans appel réseau.
- Confusion entre ce connecteur et le pipeline d'extraction
  documentaire existant (`extraction_bridge.py`) — mitigé en réutilisant
  explicitement le même pattern plutôt qu'en créant une variante
  divergente.

## Traçabilité

- RFC-0018 : Review → **Adopté** (2026-07-20, volet en ligne seulement)
- Cette décision : Proposé → **Validated** (2026-07-20)
- Tranche 1 (schéma de données) : à suivre dans cette décision au fil
  de l'implémentation.
